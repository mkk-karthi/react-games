## 🏗️ Architectural Design Patterns

The project follows a **Modular Hook & Component Architecture**, separating game logic, board styling computations, audio presets, and score state tracking from layout layers.

```
                  ┌───────────────────────┐
                  │        App.tsx        │
                  │   (Container Shell)   │
                  └───────────┬───────────┘
            ┌─────────────────┼─────────────────┐
            ▼                 ▼                 ▼
  ┌──────────────────┐┌──────────────┐┌──────────────────┐
  │   ControlPanel   ││    Board     ││    InfoPanel     │
  │ (Control/History)││  (8x8 Grid)  ││ (Scores/Captured)│
  └──────────────────┘└───────┬──────┘└──────────────────┘
                              ▼
                      ┌──────────────┐
                      │  ChessPiece  │
                      │  (Memoized)  │
                      └──────────────┘
```

### 1. State Flow & Context Setup
`App.tsx` behaves strictly as a stateless controller shell. It invokes three decoupled custom hooks:
1. **`useAudio`**: Manages Web Audio Context oscillator lifecycles, frequency configurations, and audio settings (mute/unmute toggles).
2. **`useScores`**: Coordinates score tallies for White wins, Black wins, and draws, with automatic fallback serialization to `localStorage`.
3. **`useChessGame`**: Implements the main state loop. It consumes audio triggers and score recording callbacks, maintaining chess board states, history matrices, animations, and active timers.

### 2. Rendering Optimizations
- **Gradients Singleton (`<SvgDefs>`)**: To avoid DOM bloating, the SVG wood gradients (`#whiteWood` and `#darkWood`) are defined once inside a hidden SVG element rendered by `Board.tsx`. All chess pieces reference these shared gradients dynamically.
- **Component Memoization (`React.memo`)**: `ChessPiece` is memoized to prevent expensive SVG re-renders when other squares on the board update.

## ⚓ Custom Hooks API Specifications

### `useAudio()`
Manages all game acoustics using oscillator waves without external asset dependencies.
- **State**:
  - `muted: boolean` - Sound status state.
- **API**:
  - `toggleMute() => void` - Toggles the `muted` state.
  - `playTone(preset: ToneKind) => void` - Plays a custom synthesized audio sweep according to tone types (`'start' | 'move' | 'capture' | 'end' | 'check' | 'castle' | 'promotion' | 'error' | 'select' | 'victory' | 'draw' | 'undo'`).

### `useScores()`
Handles localStorage-backed match record scoring.
- **State**:
  - `scores: StoredScores` - Record containing `{ white: number, black: number, draws: number }`.
- **API**:
  - `recordWin(color: 'w' | 'b') => void` - Increments winning score.
  - `recordDraw() => void` - Increments draw score.
  - `resetScores() => void` - Resets and clears scores in state & storage.

### `useChessGame(config)`
Contains the chess game loop, CPU move logic, and piece selectors.
- **Arguments**:
  - `playTone: (preset: ToneKind) => void`
  - `recordWin: (color: 'w' | 'b') => void`
  - `recordDraw: () => void`
- **State**:
  - `board: (Piece | null)[][]` - Current 8x8 matrix representing chess positions.
  - `selected: Square | null` - The currently active square.
  - `legalTargets: Square[]` - Allowed move targets for the selected piece.
  - `gameStarted: boolean` - True if game overlay is closed.
  - `computerThinking: boolean` - True if CPU engine calculations are processing.
  - `lastMove: { from: Square; to: Square } | null` - The coordinates of the previous move.
  - `capturedPieces: Piece[]` - Tracked collection of captured pieces.
- **API**:
  - `selectSquare(square: Square) => void` - Selected square move router.
  - `startGame(mode: GameMode) => void` - Transitions match out of the start overlay.
  - `newGame() => void` - Re-initializes board coordinates and clears active timers.
  - `undoMove() => void` - Reverts the last move (and reverts computer move if singleplayer).

## 🎨 Theme Tokens & Color Conventions

We replaced over 100 duplicate variables with **34 Design Tokens** located inside `src/styles.css`. 

### Key Rules for Colors
1. **Palette References**: Do not use hardcoded hex keys (`#fff`, `#000`) or Tailwind built-ins (like `text-amber-950`) inside component files. Always map color layers to custom tailwind properties:
   - Use `text-panel-text` instead of dark amber text values.
   - Use `text-coordinate` for board algebraic notation coordinates.
   - Use `text-no-captures` for panel empty states.
2. **Gradient Styling (`App.css`)**: Gaps and gradients must reference the base token variables (such as `var(--color-wood-300)`). If transparency layers are required, use helper variables or annotated inline `rgba()` overrides containing the original token comments:
   ```css
   rgba(255,236,178,0.90) /* wood-50 @90% */
   ```

## 🧪 Testing Guidelines & Patterns

All tests are powered by **Vitest** and **React Testing Library**. The test base covers three distinct levels:
1. **Utility Tests**: Verifying calculations for grid styles and indexing (`src/utils/__tests__`).
2. **Hook Tests**: Mocking timers and functions to assert state cycles on moves, score changes, and sound triggers (`src/hooks/__tests__`).
3. **Component Tests**: Simulating pointer events, checking screen landmarks, checking ARIA states, and testing layout responsiveness (`src/components/__tests__`).

### Adding New Tests
- Keep tests isolated.
- Mock Web Audio `AudioContext` and `OscillatorNode` globally (configured inside `src/setupTests.ts`).
- When writing tests for board clicks or scoreboards, query elements using explicit accessible roles (`role="dialog"`, `role="button"`, `aria-label`) or unique DOM IDs:
  ```typescript
  const startButton = screen.getByRole('button', { name: /start game/i });
  expect(startButton).toBeInTheDocument();
  ```
